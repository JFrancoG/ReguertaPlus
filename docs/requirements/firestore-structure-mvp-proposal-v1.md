# Firestore MVP Structure Proposal (v1)

Date: 2026-03-06

Canonical field contract:
- docs/requirements/firestore-collections-fields-v1.md

## 1. Goal

Define a Firestore structure for MVP that:
- Reuses current collections as much as possible.
- Supports confirmed business rules (orders, commitments, shifts, roles, notifications).
- Enables safe incremental evolution.

## 2. Current inventory found

Existing collections:
- `users`
- `users/*/devices` (subcollection)
- `products`
- `orders`
- `orderLines` (legacy name in current dataset)
- `containers`
- `measures`
- `news` (currently empty)

Confirmed legacy field sets:
- `containers`: `name`, `plural`
- `measures`: `abbreviation`, `name`, `plural`, `type`
- `users` (observed): `available`, `companyName`, `email`, `isAdmin`, `isProducer`, `lastDeviceId`, `name`, `numResignations`, `phone`, `surname`, `tropical1`, `tropical2`, `typeConsumer`, `typeProducer`, plus `devices` subcollection
- `products` (observed): `available`, `companyName`, `container`, `description`, `name`, `price`, `quantityContainer`, `quantityWeight`, `stock`, `unity`, `urlImage`, `userId`
- `orders` (observed): `name`, `surname`, `userId`, `week`
- `orderLines` (observed): `companyName`, `orderId`, `productId`, `quantity`, `subtotal`, `userId`, `week`

Operational config currently used:
- `develop/collections/config/global`
- `production/collections/config/global`
with:
- `cacheExpirationMinutes`
- `lastTimestamps`
- `otherConfig.deliveryDayOfWeek`
- `versions.{android,ios}.{current,min,forceUpdate,storeUrl}`

Namespace layout currently used:
- `<env>/collections/...` for current production data
- `<env>/plus-collections/...` for the new model rollout
- `<env>` in cloud: `develop`, `production`

Legacy-rich `orderlines` fields are already present and should be preserved until cleanup is proven safe.

## 3. MVP design principles

- Keep `users`, `products`, `orders`, `orderlines` as core.
- Reuse existing per-user device registry for push delivery (`users/{userId}/devices`), including the latest known `fcmToken`.
- Support pre-authorized onboarding by email with first-login UID linking.
- Formalize startup remote version policy and freshness-driven selective sync.
- Use logical delete for historical entities.
- Keep consumer display-name snapshots in `orders` and `orderlines` to support producer views/history safely.
- Keep snapshot fields in `orderlines`.
- Keep `orders.totalsByVendor` keyed by stable `vendorId` (not display names).
- Keep `orders.autoGeneratedReason` as optional trace metadata only when `isAutoGenerated == true`.
- Keep `products.productImageUrl` as canonical image source for product cards/details.
- Use `users.producerCatalogEnabled` as producer-level catalog visibility switch and keep `products.isAvailable` as product-level availability.
- Keep producer classification explicit in `users` with `producerParity` (`even`|`odd`|`null`) and `isCommonPurchaseManager` (boolean), instead of adding extra role values.
- Keep `products` season-agnostic; yearly or campaign-specific seasonality belongs in `seasonalCommitments`, with a separate campaign entity only if later needed.
- Use hybrid calendar resolution: default day in `config/global.deliveryDayOfWeek` + week overrides in `deliveryCalendar/{weekKey}`.
- Keep schema extension points for weighted products (`pricingMode = weight` with single `price`) to avoid future breaking migrations.
- Use explicit eco-basket option values (`pickup`, `no_pickup`) on order lines instead of free-text naming conventions.
- Avoid over-normalization in MVP.
- Add new collections only for missing critical capabilities.

## 4. Proposed model

### 4.1 Reused collections (expanded)

- `users/{userId}`
- `users/{userId}/devices/{deviceId}` (subcollection)
- `products/{productId}`
- `orders/{orderId}`
- `orderlines/{orderlineId}` (mapped from legacy `orderLines` where needed)

### 4.2 New minimal collections

- `deliveryCalendar/{weekKey}`
- `seasonalCommitments/{commitmentId}`
- `sharedProfiles/{userId}`
- `shifts/{shiftId}`
- `shiftSwapRequests/{requestId}`
- `news/{newsId}`
- `notificationEvents/{eventId}` (recommended)
- `config/global` is required per environment for startup/version/sync policy:
  - current path: `<env>/collections/config/global`
  - rollout path: `<env>/plus-collections/config/global`
  - `<env>`: `develop` or `production`

Shift/source semantics:
- `shifts.source` uses `app` when managed from app/admin flows.
- `shifts.source` uses `google_sheets` when coming from Sheets integration/sync.

Notification event contract:
- `title`: string
- `body`: string
- `type`: `order_reminder` | `order_auto_generated` | `shift_swap_requested` | `shift_swap_accepted` | `shift_swap_applied` | `shift_updated` | `news_published` | `admin_broadcast`
- `target`: `all` | `users` | `segment`
- `targetPayload`:
  - `all`: empty/null payload
  - `users`: `{ userIds: string[] }`
  - `segment`: `{ segmentType, ... }` where `segmentType` is one of:
    - `members_with_pending_order` (requires `weekKey`)
    - `users_with_shift` (requires `shiftId`)
    - `producers_by_vendor` (requires `vendorId`)
    - `role` (requires `role`: `member` | `producer` | `admin`)
- `sentAt`: timestamp
- `createdBy`: `system` or authorized `userId`
- `weekKey`: optional, only when the event applies to a concrete week

Delivery calendar scope:
- `deliveryCalendar` stores exception weeks only (doc ID = `weekKey`).
- Missing week document means fallback to `config/global.deliveryDayOfWeek`.

## 5. Core relationships

- users 1..N orders
- orders 1..N orderlines
- users 1..1 sharedProfiles
- users 1..N seasonalCommitments
- users 0..1 auth identity link (`users.authUid`) after first authorized login
- users 1..N devices (`users/{userId}/devices/{deviceId}`)
- users 0..1 last active device pointer (`users.lastDeviceId`)
- shifts 1..N shiftSwapRequests

## 6. Suggested indexes (minimum)

- `orders`: `(userId ASC, weekKey DESC)`
- `orders`: `(weekKey ASC, consumerStatus ASC)`
- `orderlines`: `(orderId ASC, companyName ASC)`
- `orderlines`: `(vendorId ASC, weekKey DESC)`
- `products`: `(vendorId ASC, archived ASC, isAvailable ASC)`
- `users/{userId}/devices`: `(lastSeenAt DESC)` (if recency listing is required)
- `shifts`: `(date ASC, type ASC)`
- `shiftSwapRequests`: `(targetUserId ASC, status ASC, requestedAt DESC)`
- `seasonalCommitments`: `(userId ASC, seasonKey ASC, active ASC)`

## 7. Security model summary

- `users`: admin writes for lifecycle/roles; first authorized login can link `authUid` under controlled rules.
- `users/{userId}/devices`: authenticated user self-write for own device metadata and token refresh; admin read/support access.
- `config/global`: authenticated read; controlled write (admin/system) for version/sync policy.
- `sharedProfiles`: authenticated member read; owner write.
- `products`: producer-owner or admin write; `vendorId` immutable.
- `orders/orderlines`: owner access + producer scoped access + admin support access.
- `news`: admin-only write.
- `shifts/shiftSwapRequests`: member read; controlled write transitions.
- authenticated accounts without matching authorized member email remain in restricted mode (no operational writes).

## 8. Incremental migration strategy

Phase 1:
- Keep current collections.
- Add optional fields gradually, including `users.emailNormalized`, nullable `users.authUid`, `users.lastDeviceId`, `users.producerParity`, `users.isCommonPurchaseManager`, `products.unitAbbreviation`, and `products.packContainerAbbreviation`.
- Keep/align `users/{userId}/devices/{deviceId}` as canonical source for per-device metadata.
- Keep/align environment-scoped `config/global` contract based on current fields:
  - `cacheExpirationMinutes`
  - `lastTimestamps`
  - `otherConfig.deliveryDayOfWeek` (with normalized top-level `deliveryDayOfWeek` as target)
  - `versions.{android,ios}.{current,min,forceUpdate,storeUrl}`

Phase 2:
- Introduce new collections (`deliveryCalendar`, `seasonalCommitments`, `sharedProfiles`, `shifts`, `shiftSwapRequests`, `news`).
- Add commitment validations.
- Include eco-basket `no_pickup` as a paid commitment-valid option on order lines.
- Enforce single eco-basket price across parity producers and pickup options (`pickup`/`no_pickup`).

Phase 3:
- Add Sunday reminders (20:00, 22:00, 23:00 Europe/Madrid).
- Optional auto-order generation for forgotten commitments (excluding members with `isActive = false`).
- Route reviewer account to develop backend by allowlist.

Phase 4 (post-MVP catalog extension):
- Enable weighted products with `pricingMode = weight`.
- Store weighted-order snapshots in `orderlines` (`pricingModeAtOrder`, `priceAtOrder`, `quantity` in product weight unit).

## 9. Compatibility with existing work

Direct reuse:
- Existing order/orderline history stays valid.
- Existing users/products remain core entities.
- Existing `users/{userId}/devices` subcollection remains valid and is formally incorporated in the contract.
- Existing config timestamps remain in place.

Required additions:
- New collections for missing features only.
- Security/index updates aligned with role model.

## 10. Technical risks to monitor

- Producer data access scoping in aggregated views.
- Google Sheets sync conflict handling.
- Reviewer routing robustness in production app.
- Reminder reliability when apps are backgrounded.
- Device metadata freshness and stale `lastDeviceId` references.
- Decimal rounding consistency for weighted calculations (`quantity * price`) across Android, iOS, and backend.
